Designing Multi-Tenant SaaS from Scratch

When you set out to build a multi-tenant SaaS platform, the first question isn't about technology — it's about isolation. How much do your tenants need? The answer shapes everything from your database strategy to your deployment model.

The Isolation Spectrum

At one extreme, you have fully isolated deployments — separate databases, separate app instances, maybe even separate infrastructure per tenant. At the other, you have shared-everything — one database, one app, tenant separation purely through application logic. Most enterprise systems land somewhere in the middle.

For our platform, we chose shared infrastructure with logical isolation: one database with row-level tenant filtering, shared application services with tenant context injected via middleware, and tenant-specific configuration stored in a separate management plane.

CQRS Made It Possible

The CQRS pattern was the architectural decision that made multi-tenancy manageable. By separating read and write models, we could optimize each side independently. Writes go through a strict command pipeline with tenant validation at every step. Reads use denormalized views that are pre-filtered by tenant, making queries fast without complex join logic.

MediatR serves as the backbone — every command and query flows through a pipeline where tenant context, authorization, and audit logging happen automatically. New features don't need to think about tenancy; it's handled by the infrastructure.

The best multi-tenant architecture is one where developers building features don't need to think about multi-tenancy.

What I'd do differently next time: invest earlier in tenant-level observability. When something breaks in a shared system, the first question is always "which tenant is affected?" Having metrics and logs tagged with tenant context from day one would have saved us weeks of debugging.

Optimizing LLM API Token Costs in Production

When we first integrated AI into our enterprise platform, our API costs were terrifying. We were burning through tokens like they were free. Six months later, we'd cut costs by 60% without any degradation in quality. Here's how.

The Three-Lever Framework

Cost optimization for LLM APIs comes down to three levers: which model you call, how much context you send, and how often you call it.

Model switching was the biggest win. Not every request needs the most capable model. We built a routing layer that evaluates the complexity of each request and routes simple ones to smaller, cheaper models. Classification, extraction, and reformatting go to a fast model. Complex reasoning, multi-step analysis, and creative generation go to the flagship.

Prompt Caching Changed Everything

Prompt caching was our second biggest lever. In our system, the system prompt includes extensive context about the platform, the tenant's configuration, and the user's permissions. This context doesn't change between requests from the same tenant. By structuring our prompts so the static context comes first, we get cache hits on 70-80% of tokens for subsequent requests.

// System prompt structure for cache optimization
// [STATIC] Platform context + tenant config (~2000 tokens) ← cached
// [SEMI-STATIC] User permissions + recent history (~500 tokens)
// [DYNAMIC] Current request (~100-300 tokens)

The key insight: token order matters. Put your most stable content at the beginning of the prompt, and your most dynamic content at the end.

My Agentic AI Learning Journey

Six months ago, I couldn't have told you the difference between a chatbot and an AI agent. Today, I'm building multi-agent systems in C# that autonomously handle development tasks. Here's the path I took, and what I wish I'd known from the start.

Start with RAG, Not Agents

Everyone wants to jump straight to agents, but RAG (Retrieval-Augmented Generation) is the foundation. It's the difference between an AI that hallucinates and one that gives answers grounded in your actual codebase and documentation. I started with embedding our internal docs and code comments into a vector store and building a retrieval pipeline.

The Agent Loop: Where It Gets Interesting

An agent loop is deceptively simple: observe → think → act → observe. The complexity comes from deciding when to stop, how to handle failures, and how to maintain context across multiple tool calls. In .NET, I built this as a state machine with MediatR handling each step as a command.

The best way to understand AI agents is to build one that fails — then figure out why.

I'm documenting the full journey here on this blog. Next up: how MCP servers changed my understanding of what AI tools can actually do in production.
