Design Twitter / X: Timeline Fanout at Scale

The fanout problem — how do you push a tweet to 50 million followers in under 5 seconds?

The Problem

You're in a system design interview. The interviewer asks you to design a Twitter-like platform that handles 500M tweets per day, 300M active users, and 150M daily active users. The system must generate home timelines with P99 latency under 200ms.

Focus on timeline generation — this is where most candidates stumble. How do you handle a tweet from a user with 30 million followers without taking down the system? How do you store and serve timelines at 1.15 million reads per second? What is the trade-off between write-time fanout and read-time aggregation?

Solution Approach

Start with the core data model: users, tweets (~300 bytes each), follower graph, and home timelines. At 500M tweets/day you are looking at ~15K write QPS and 1.15M timeline read QPS — reads dominate by 75x, so you optimize for reads at the cost of write complexity.

The critical decision is timeline generation strategy. A naive pull model (aggregate on read) breaks immediately — querying 2,000 followed accounts per request means 2,000 DB lookups for every timeline view. A pure push model (fanout on write) works until a celebrity with 30M followers tweets, triggering 30M simultaneous Redis writes.

Twitter's answer: hybrid fanout. For regular users, fanout on write — when a tweet is posted, drop it onto Kafka, let fanout workers asynchronously push the tweet ID into each follower's Redis timeline cache (a sorted set capped at ~800 entries). For heavy hitters above a follower threshold, skip fanout entirely. At read time, merge their latest tweets into the requesting user's pre-built cache. O(1) reads for 99.9% of users; the small celebrity merge cost is predictable and bounded.

Data layer: tweet content in a distributed NoSQL store sharded by tweet ID, social graph in a separate service, full-text search via Elasticsearch. Twitter reported this hybrid approach delivered a 5x improvement in timeline load time over pure pull-on-read aggregation.

In interviews, the interviewer is testing whether you recognize the power-law distribution of follower counts. Mention the threshold approach — Twitter reportedly uses ~1M followers — and justify why you chose that boundary.

Trade-offs

When to use

  • Social feeds with power-law follower distribution (a few accounts have millions of followers)
  • Systems where read QPS dwarfs write QPS by 10x or more
  • Fan-out use cases where per-user inbox/timeline SLA latency is critical

When to avoid

  • Graphs where follower counts are uniformly small — full push-on-write is simpler and sufficient
  • Systems with extremely high celebrity post frequency where even read-time merge becomes expensive
  • Applications where strict timeline ordering is required — async fanout introduces ordering complexity
Pattern: Hybrid Fanout (Push/Pull)

Write-time fanout to Redis timeline caches handles the common case, while read-time aggregation for power-users prevents thundering-herd write amplification. The same pattern appears in enterprise notification systems, multi-tenant event feeds, and SaaS activity streams — whenever your graph has a power-law distribution, a pure push or pure pull strategy fails at one extreme. The hybrid acknowledges the data distribution reality and handles each case with the optimal strategy, at the cost of a more complex read path that must detect and merge heavy-hitter content.
