Rate Limiting in Multi-Tenant APIs

How do you enforce fair API usage across hundreds of tenants without degrading performance for well-behaved clients?

The Problem

You're running a multi-tenant SaaS platform where hundreds of tenants share the same API infrastructure. One tenant starts hammering your endpoints with 10x their normal traffic — maybe a runaway batch job, maybe a misconfigured integration.

Your other tenants are starting to see increased latency. You need a rate limiting strategy that’s fair, per-tenant, and doesn’t require a database call on every single request.

Solution Approach

The proven approach is a distributed token bucket backed by Redis, with tenant-specific limits derived from their subscription tier. Each API request extracts the tenant ID from the JWT or API key, then checks a Redis key like ratelimit:{tenant_id}:{window}.

Use a sliding window counter rather than fixed windows to avoid the burst-at-boundary problem. Redis’s MULTI/EXEC lets you atomically increment the counter and set TTL in one round-trip. For .NET, the System.Threading.RateLimiting namespace gives you TokenBucketRateLimiter, SlidingWindowRateLimiter, and ConcurrencyLimiter out of the box.

The key architectural decision: implement rate limiting as middleware, not in individual controllers. This keeps it consistent and lets you layer limits — a global limit per tenant, plus endpoint-specific limits for expensive operations like report generation or bulk exports.

Stripe does this well with a two-tier approach: a generous per-second limit for normal bursts, and a stricter per-minute limit for sustained abuse. Both are communicated via X-RateLimit-* headers so clients can self-throttle.

Trade-offs

When to use

  • Multi-tenant APIs with shared infrastructure
  • Public-facing APIs with unpredictable traffic
  • Services where one tenant can impact others

When to avoid

  • Internal service-to-service calls (use circuit breakers instead)
  • Low-traffic admin APIs where limits add unnecessary complexity
  • Real-time streaming where latency of Redis check matters
Pattern: Token Bucket

Tokens accumulate at a fixed rate and are consumed per request. This naturally handles bursts (bucket can fill up during quiet periods) while enforcing average throughput. In multi-tenant SaaS, assign different bucket sizes per subscription tier — your enterprise tenants get a bigger bucket than free-tier users.
